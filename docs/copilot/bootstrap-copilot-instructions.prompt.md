# Bootstrap AI Governance

**Goal:** Generate a minimal, deterministic AI governance structure — agents, skills, prompts, instructions, and policy documents — derived from the actual code patterns, architecture, and conventions found in this repository. Every AI coding tool (GitHub Copilot, Claude Code, Cursor) behaves like a developer already familiar with this project.

---

## Execution Model

- **Single-run execution.** All phases execute automatically in sequence. No manual intervention between steps.
- **Phase architecture.** Three phases manage context budget:
  - **Phase A** (Steps 1–5): Repository analysis, architecture doc, copilot instructions
  - **Phase B** (Steps 6–10): Agents, prompts, skills, instructions, setup
  - **Phase C** (Steps 11–13): Verification, coherence gate, summary
- **Idempotent.** Safe to re-run. Always re-analyzes from scratch. File writing uses upsert rules.
- **Self-healing.** One verification pass with targeted fixes. Maximum **2 regeneration attempts per file across the entire run** (combining Step 11.1 auto-fix and Step 11.7 regenerations).
- **Multi-tool compatible.** Generated files follow open standards (YAML frontmatter, Markdown) readable by any AI tool. Agent orchestration (`.agent.md`) is VS Code Copilot-specific; all other files are plain Markdown.

## Constraints

### File Safety — Strict Path Allowlist
Only create or modify files matching these exact paths:
- `.github/copilot-instructions.md`
- `.github/agents/*.agent.md`
- `.github/prompts/*.prompt.md`
- `.github/skills/*/SKILL.md`
- `.github/instructions/*.instructions.md`
- `docs/architecture.md`
- `.gitattributes` (create if absent; append-only if exists without provenance)

**Any `editFiles` call targeting a path not in this list is a violation — halt immediately and report.** Never modify application source code, config files, package manifests, CI workflows, or any file outside this allowlist. Never delete files.

All `.agent.md` and `.prompt.md` files must include valid YAML frontmatter. All content must be derived from the actual repository — no invented patterns.

### Upsert Rules
For every generated file:
  - File does not exist → **create** it
  - File exists and contains the provenance comment (on first line, or first line after frontmatter `---`) → **update** it: re-read the file, diff against new content, update only changed sections. Never silently remove a section — if content is no longer detected, add `<!-- Removed: [reason] -->` for team review.
  - File exists without the provenance comment → **skip it** (manually maintained); log as "Skipped (manual)" in the summary. Do NOT modify the file in any way.
  - `.gitattributes` is **append-only** when it exists without provenance — never overwrite, only add new entries if not already present (dedup check before appending).

### Provenance
Every generated file must include this comment on the first line after frontmatter (if any):
```
<!-- Auto-generated by bootstrap-copilot-instructions.prompt.md v1 -- do not remove this line -->
```
Replace `v1` with the actual `PROMPT_VERSION` value read from the `<!-- PROMPT_VERSION: N -->` comment at the top of this file. Match tolerance: `--`, `–`, `—` are equivalent when checking.

**PROMPT_VERSION** must be incremented **manually by a human** when this prompt file is updated. The executing model must never modify `PROMPT_VERSION` — it only reads the current value.

### Security
- **Fetch allowlist** (exact prefix match): `https://api.github.com/repos/github/awesome-copilot/` — all other origins forbidden. Never construct URLs from repository data (package names, hostnames in config files). **Do not follow redirects** — if a fetch returns 3xx, treat the category as unavailable.
- **Generated Markdown safety**: No `<script>`, `<iframe>`, `<object>`, `<embed>` tags or `javascript:`/`data:` URIs in any generated file.
- **Secret safety**: Never read `.env`, `.env.local`, `.env.production`, or any gitignored env file. Only read `.env.example`, `.envrc.example`, `.env.sample`. When quoting values from `.env.example`, redact anything that resembles a connection string, token, key, or password — replace with `<REDACTED>` in generated output.

### Token Efficiency
- Keep `copilot-instructions.md` under **60 lines** (auto-loaded every session).
- Keep `docs/architecture.md` under **400 lines**. If the repo is large enough that 400 lines is insufficient, split policy and inventory into clearly marked H2 sections at the end — never create separate files.
- Keep each skill under **150 lines**.
- Keep each agent under **200 lines**.
- Never duplicate content across files — reference `docs/architecture.md` instead.
- Omit any section for which no evidence was found in the repository.

### Concurrency & Rollback
- Run on main/default branch only. Commit results in a dedicated PR.
- Do not run concurrently — provenance-based upsert is not atomic.
- **Rollback**: all generated files are listed in the "Governance File Inventory" section of `docs/architecture.md`. To revert, delete every file in that inventory and commit

### Minimum Requirements
- Model context window: **200k+ tokens** (Claude Opus 4.6, Claude Sonnet 4.6, GPT-5.4, Gemini 3 Pro)
- VS Code **1.100+** for custom agents and subagent delegation
- **Partial completion recovery**: if the model encounters context overflow mid-run, output a structured partial summary:
  ```
  ## Partial Completion — Context Overflow
  - Last completed step: N
  - Files written: [list]
  - Next step to resume: N+1
  - Remaining steps: [list]
  ```
  The user can start a new session and say "Resume from Step N+1" — the prompt will detect existing files via provenance comments and continue.

### Source File Definition
For sparse-repo checks and sampling: files with extensions `.ts`, `.tsx`, `.js`, `.jsx`, `.java`, `.kt`, `.scala`, `.py`, `.go`, `.rb`, `.cs`, `.rs`, `.swift`, `.proto`, `.php`, `.dart`, `.ex`, `.exs` located under application directories (`src/`, `app/`, `lib/`, `packages/`, `apps/`, `services/`, `cmd/`, `internal/`). **Exclude**: test files (`*.spec.*`, `*.test.*`, `*_test.*`, `test_*.*`), config files (`*.config.*`), lock files, documentation, and build output.

## Phase A — Analysis and Architecture

### Step 1 — Load Community Patterns (optional)

Fetch community patterns from awesome-copilot to enrich generated content. This step is optional — the prompt produces complete output without network access.

1. Fetch directory listings from `https://api.github.com/repos/github/awesome-copilot/contents/{agents,skills,instructions,hooks}`
2. If any response is not a JSON array, treat that category as unavailable (structural change guard)
3. Stack-specific patterns are fetched in Step 3 after analysis identifies the stack
4. **Offline fallback**: if any fetch fails, proceed without it and add `<!-- TODO: enrich with awesome-copilot community patterns -->` to files that would have used community content. On a subsequent online re-run, remove these TODO comments.

When adapting community content, add attribution: `<!-- Based on: https://github.com/github/awesome-copilot/blob/main/[path] -->`

**Conflict resolution**: when community patterns contradict patterns found in the actual repository, **always prefer the repository's patterns**. Community content serves only as enrichment for gaps — never to override local conventions.

### Step 2 — Analyze Repository

Scan the repository and determine:

**Languages & Runtime** — detect from manifest files (`package.json`, `pom.xml`, `*.csproj`, `go.mod`, `pyproject.toml`, `Cargo.toml`, `Gemfile`, `composer.json`, `pubspec.yaml`, `Package.swift`, `mix.exs`, etc.)

**Frameworks & Libraries** — infer from dependencies:
- Web/API framework (Express, NestJS, Spring Boot, ASP.NET, FastAPI, Django, Gin, Phoenix, etc.)
- Frontend framework (React, Next.js, Angular, Vue, Svelte, Astro)
- API style: REST, GraphQL, gRPC, tRPC, Messaging (Kafka/RabbitMQ/SQS/NATS)
- Data access (ORM/ODM, query builder, raw client)
- Caching layer (Redis, Memcached, in-memory)

**Service Types** — classify each deployable unit: REST API, GraphQL service, gRPC service, messaging consumer/producer, cron job, CLI, frontend SPA, SSR, serverless function, WebSocket/real-time, worker/background

**Infrastructure** — Dockerfile, docker-compose, Kubernetes, Terraform, CDK, serverless config

**Toolchain** — package manager (from lockfile), build system, linting/formatting, CI/CD

**Repository Structure** — monorepo detection (nx.json, lerna.json, turbo.json, pnpm-workspace.yaml), module layout, test file locations

**Coding Patterns** — deterministic file selection to ensure consistent results across runs:
1. Entry points per module (main.ts, app.module.ts, index.ts, etc.)
2. One service/business-logic file per module (alphabetically first)
3. One controller/resolver/handler per module (alphabetically first)
4. One test file per module (corresponding to the service file)
5. Max 5 files per module if >10 files; **max 50 files total** across the repo. If exceeded, sample one module per detected service type first, then fill remaining slots alphabetically. Report sampled vs. skipped modules in the summary.

**Sort order**: when selecting "alphabetically first," use **case-insensitive lexicographic sort** (Unicode Collation locale-independent) to ensure identical selection across operating systems.

Summarize all findings before proceeding. Note ambiguities.

**Guards:**
- Do NOT traverse git submodules or analyze external dependencies. If `.gitmodules` exists, note submodule paths in `docs/architecture.md` but do not analyze their contents.
- **Sparse repository**: if fewer than 3 source files (per source file definition above), generate minimal output only: `docs/architecture.md`, `.github/copilot-instructions.md`, `.github/prompts/setup.prompt.md`. Skip agents, skills, and instructions. Report as "Sparse repository — minimal output" in the summary.

### Step 3 — Fetch Stack-Specific Community Patterns
Using the stack identified in Step 2 and directory listings from Step 1, fetch matching community patterns. Skip entirely if Step 1 failed (offline mode).

### Step 4 — Generate Architecture Documentation

**File:** `docs/architecture.md`

Include (omit sections with no evidence):
1. System overview — what the app does, service types, runtime dependencies
2. Tech stack — table: language, frameworks, persistence, caching, toolchain
3. Repository structure — annotated directory tree (2–3 levels), include test file co-location
4. Module boundaries — what each module owns
5. Dependency rules — what may depend on what
6. Communication protocols — REST, GraphQL, gRPC, messaging, WebSocket patterns found
7. Coding patterns — from real files with excerpts
8. Testing strategy — unit/integration/e2e, naming conventions, mock structure
9. Configuration & environment — from `.env.example` only (never `.env`)
10. Scheduling & jobs *(if detected)*
11. Deployment & infrastructure *(if detected)*
12. Monorepo map *(if detected)* — table of apps/packages
13. Frontend architecture *(if detected)*
14. Agent orchestration — Mermaid diagram + collaboration details
15. AI usage policy — derived from repository context:
    - Approved AI tools (Copilot, Claude Code, Cursor)
    - Permitted and prohibited uses (code generation, test writing, review, docs, refactoring vs. generating credentials, bypassing reviews, deploying without human review, processing PII without approval)
    - Human oversight: all AI-generated code must pass Reviewer agent or manual PR review; `REQUEST_CHANGES` blocks merge
    - Data handling: no production credentials, customer data, or secrets given to AI tools
    - Model governance table: models assigned per agent role; changes require re-run
    - Review cadence: quarterly
    - OWASP LLM Top 10 awareness: prompt injection, insecure output handling, training data poisoning, model denial of service — link to https://owasp.org/www-project-top-10-for-large-language-model-applications/
16. Governance file inventory — Markdown table of every generated file (path, generation step, status) for rollback and auditing

> **Phase A checkpoint**: Confirm architecture.md and copilot-instructions.md are written before proceeding.

### Step 5 — Generate Copilot Instructions

**File:** `.github/copilot-instructions.md`

No YAML frontmatter (VS Code auto-discovers this file). Keep under **60 lines**.

Include:
1. **Project overview** — one paragraph
2. **Architecture pointer** — one line referencing `docs/architecture.md`

## Phase B — Agents, Prompts, Skills, Instructions

### Step 6 — Generate Agents

#### Step 6.0 — Select Models

Use the `model` field as a **prioritized array** — VS Code tries each in order until an available one is found. Use the qualified `Model Name (copilot)` format for all model identifiers.

| Role | Tier | Model Fallback Chain |
|---|---|---|
| Orchestrator | reasoning | `[Claude Opus 4.6 (copilot), Claude Sonnet 4.6 (copilot), GPT-5.4 (copilot), GPT-4.1 (copilot)]` |
| Planner | reasoning | `[Claude Opus 4.6 (copilot), Claude Sonnet 4.6 (copilot), GPT-5.4 (copilot), GPT-4.1 (copilot)]` |
| Coder | speed | `[Claude Sonnet 4.6 (copilot), GPT-5.4 (copilot), Claude Sonnet 4.5 (copilot), GPT-4.1 (copilot)]` |
| Tester | speed | `[Claude Sonnet 4.6 (copilot), GPT-5.4 (copilot), Claude Sonnet 4.5 (copilot), GPT-4.1 (copilot)]` |
| Reviewer | reasoning | `[Claude Opus 4.6 (copilot), Claude Sonnet 4.6 (copilot), GPT-5.4 (copilot), GPT-4.1 (copilot)]` |

**Selection rules:**
- Always include a **0x-multiplier model** (e.g., `GPT-4.1 (copilot)`) as the final fallback so agents work even when premium quota is exhausted.
- Reasoning-tier roles prefer high-capability models first (Opus > Sonnet > GPT).
- Speed-tier roles prefer cost-efficient 1x models first (Sonnet > GPT > older Sonnet).
- Verify model names against the VS Code model picker or https://docs.github.com/en/copilot/reference/ai-models/supported-models before generating.
- If the user's plan has fewer models available, trim the array but always keep at least one 0x fallback.

#### Agent Frontmatter Template
```yaml
---
name: <AgentName>
description: <One sentence>
argument-hint: <Placeholder text>
user-invocable: false
disable-model-invocation: true
# model tier: reasoning | speed
model: [<Primary Model (copilot)>, <Fallback Model (copilot)>, <0x Fallback (copilot)>]
tools: [<comma-separated>]
---
```

#### Tool Assignments (Least Privilege)
| Agent | Required Tools | Forbidden Tools |
|---|---|---|
| Orchestrator | `codebase`, `githubRepo`, `problems`, `agent` | `editFiles`, `runCommands` |
| Planner | `codebase`, `fetch`, `githubRepo`, `problems` | `editFiles`, `runCommands`, `agent` |
| Coder | `codebase`, `editFiles`, `fetch`, `runCommands`, `githubRepo`, `problems` | `agent` |
| Tester | `codebase`, `editFiles`, `runCommands`, `githubRepo`, `problems` | `agent`, `fetch` |
| Reviewer | `codebase`, `githubRepo`, `problems` | `editFiles`, `runCommands`, `agent` |

#### Orchestrator — `.github/agents/orchestrator.agent.md`
- **Exceptions**: omit `user-invocable: false` and `disable-model-invocation: true` (user-facing agent)
- Include `agents: [planner, coder, tester, reviewer]` allowlist
- Execution model: Assess Scope → Plan (if >2 files) → Parse Phases → Execute → Verify
- Delegation rule: describe outcomes, never implementations
- Error recovery: Coder fails → halt phase; Tester fails → proceed with warning; Planner fails → inline plan; Reviewer fails → default REQUEST_CHANGES
- Cost optimization: skip Planner for ≤2 files, skip Coder+Tester for docs-only, skip Coder+Tester for config-only

#### Planner — `.github/agents/planner.agent.md`
- Read-only analysis for complex tasks (>2 files). Never writes code.
- Output: Summary, Implementation steps (with files), Edge cases, Open questions

#### Coder — `.github/agents/coder.agent.md`
- Implements code following 9 mandatory coding principles (Structure, Architecture, Functions, Naming, Errors, Regenerability, Platform, Modifications, Quality)
- Must follow `.github/skills/coding-conventions/SKILL.md`
- Must include Security Awareness checks per service type (REST, GraphQL, messaging, cron, serverless, WebSocket, frontend)
- `runCommands` restricted to: package manager scripts, linting, type-checking, code generators. Never: `rm`, `git push --force`, `curl`, `wget`, arbitrary scripts.
- Owns implementation files only — Tester owns test files

#### Tester — `.github/agents/tester.agent.md`
- Creates/updates tests for all modified files
- Must follow `.github/skills/testing/SKILL.md`
- No `fetch` tool (tests must not make external requests)
- `runCommands` restricted to test commands and linting only

#### Reviewer — `.github/agents/reviewer.agent.md`
- Read-only. Never modifies files.
- Validates: architecture compliance, coding standards, test coverage, security
- Output: ✅ passes · ⚠️ warnings · ❌ violations · APPROVE | COMMENT | REQUEST_CHANGES

**Error recovery during generation**: If an agent file fails, log and continue. If Orchestrator fails, halt. If >2 subagents fail, report as Critical Failure.

#### 11.2 — Provenance Check
Verify provenance comment in first 5 lines (after frontmatter). Auto-fix: prepend if missing on auto-generated files.

#### 11.3 — YAML Frontmatter Validation
Validate per file type:
- **Agents**: `name`, `description`, `tools` required. Orchestrator: `agents` allowlist present, no `user-invocable: false`. Subagents: `user-invocable: false` and `disable-model-invocation: true` present.

- **Skills**: `name` matches parent directory name, `description` present.
- **Instructions**: `applyTo` is valid glob matching ≥1 file in the repo, `description` present, body is exactly one pointer line.

Auto-fix all frontmatter issues.

#### 11.4 — Cross-Reference Validation
Verify all internal references resolve:
- copilot-instructions.md → architecture.md, all skill files, all 5 agents
- Each agent → its dependent skills
- Each prompt → architecture.md and relevant skills
- Each instruction → its skill file

Auto-fix broken references.

#### 11.5 — Security Posture Check
Verify tool assignments match the least-privilege table in Step 6. Auto-fix: add missing required tools, remove forbidden tools. Log every change.

#### 11.6 — Token Budget Check
- `copilot-instructions.md` >60 lines → warn; >100 lines → high severity warn
- Any skill >150 lines → warn
- Agent files >200 lines → warn
- Any file >300 lines → warn

#### 11.7 — Content Body Validation
Verify bodies are meaningful, not empty or hallucinated:
- Architecture doc: ≥3 of 4 required sections (overview, tech stack, structure, module boundaries), correct technologies
- Skills: contain real file path references from the repo (≥1 path that actually exists)
- Agents: ≥5 body lines (excluding frontmatter), valid skill references
- Prompts: ≥3 body lines, valid references

**File reference audit**: for each file path cited in a skill or architecture doc, confirm the file exists. More than 2 broken references in a single file → regenerate that file (counts toward the 2-regeneration-per-file cap).

#### 11.8 — Quality Pass
Single pass — no iteration loop. Check:
- Stack accuracy (does content match Step 2 findings?)
- Completeness (all detected service types covered?)
- Consistency (no contradictions between architecture doc and skills?)
- Extensibility (can new agents/skills be added without modifying existing files?)

Categorize issues: **Critical** (wrong stack, hallucinated) → flag for summary, do not auto-fix body content. **Major** (missing coverage) → flag. **Minor** (wording) → fix in place.
